# Интеграция с Telethon-проектами в mm-системе

Система `mm` поддерживает автоматическое обнаружение проектов, использующих библиотеку [Telethon](https://docs.telethon.dev/), и интегрирует в базу знаний проекта лучшие практики по безопасной работе с Telegram-аккаунтами.

Telegram применяет строгие алгоритмы для выявления автоматизации. Неправильный запуск купленного аккаунта (без прокси или с неверным фингерпринтом устройства) почти гарантированно приводит к моментальному бану. Данная интеграция помогает ИИ-агентам соблюдать дисциплину и избегать блокировок.

---

## Как это работает

### 1. Автоопределение стека (Stack Auto-detection)
Во время инициализации или обновления проекта через команду `/mm new` (скилл [mm-init-project](file:///C:/Users/louise/Desktop/louise-skills/skills/mm-init-project/SKILL.md)) система сканирует манифест-файлы проекта (`requirements.txt`, `pyproject.toml`, `package.json` и др.). 
Если среди зависимостей обнаружен пакет `telethon`, проект помечается флагом `telethon_project = true`.

### 2. Копирование инструкций по сессиям
В Obsidian Vault проекта по пути `knowledge/integrations/telethon-sessions.md` копируется файл-инструкция [telethon-sessions.md](file:///C:/Users/louise/Desktop/louise-skills/templates/telethon-sessions.md), содержащий:
* Разбор маппинга полей JSON-параметров в конструктор `TelegramClient`.
* Правильные примеры настройки ротируемого и sticky SOCKS5-прокси.
* Функцию `connect_and_validate` для разделения живых, замороженных (frozen) и забаненных аккаунтов.
* Обработку RPC-ошибок (таких как `FloodWaitError` и `AuthKeyDuplicatedError`).
* Скрипт автоматического исправления старых SQLite схем сессий.

В главном индексе базы знаний `00-home/index.md` автоматически создается ссылка на эту инструкцию.

### 3. Запись жестких ограничений в passport.md
В файл `passport.md` проекта в **Раздел 8. Контекст для промптов ⚠️** автоматически добавляются следующие правила, обязательные для проверки при генерации любого промпта:
* *При написании кода для Telethon-аккаунтов строго следуй правилам из [[telethon-sessions|инструкции по сессиям Telethon]].*
* *Запрещены параллельные запуски сессий с одним файлом .session (риск AuthKeyDuplicatedError).*
* *Все параметры устройства (fingerprint) бери строго из соответствующего .json файла.*
* *Используй SOCKS5-прокси с rdns=True для всех подключений.*

### 4. Внедрение правил в Project Instructions
В генерируемый файл `00-home/project-instructions.md` (инструкции для claude.ai Project) добавляется блок:
> *Этот проект использует Telethon. При написании кода по работе с сессиями/аккаунтами руководствуйся правилами из [[knowledge/integrations/telethon-sessions.md]].*

## Диагностика и автоисправление (`/mm check` / `mm-doctor`)

Если в проекте обнаружен Telethon, команда диагностики системы [mm-doctor](file:///C:/Users/louise/Desktop/louise-skills/skills/mm-doctor/SKILL.md) (запускаемая через `/mm check` или `/mm-doctor`) автоматически проводит статический анализ кода проекта на наличие критических уязвимостей и рисков блокировки аккаунтов.

### Выполняемые проверки:
1. **Наличие инструкций**: Проверяется, скопирован ли файл `knowledge/integrations/telethon-sessions.md` в Obsidian Vault проекта.
2. **Использование прокси**: Сканируются вызовы `TelegramClient(...)` во всех `.py` файлах. Если клиент инициализируется без прокси, выводится предупреждение.
3. **Фингерпринт устройства**: Проверяется, передаются ли параметры `device_model`, `system_version`, `app_version` динамически (например, из JSON). Жестко закодированные строки или отсутствие параметров вызовут предупреждение.
4. **Проверка на заморозку (freeze)**: Проверяется, есть ли в логике подключения вызовы дешевых методов вроде `get_dialogs(limit=1)` для быстрой детекции заморозки.
5. **Обработка исключений**: Проверяется наличие блоков обработки критических ошибок MTProto, таких как `FloodWaitError` и `AuthKeyDuplicatedError`.

### Автоисправление (Auto-fix):
Если обнаружены нарушения, `mm-doctor` предложит (с подтверждением пользователя):
* Создать готовый хелпер-модуль `telegram_builder.py` с безопасным конструктором клиента на базе параметров устройства.
* Переписать опасные вызовы `TelegramClient` на вызов безопасного построителя.

## Преимущества модульного подхода
1. **Универсальность**: Интеграция не раздувает ядро `mm` и не требует отдельной ветки. Если проект не использует Telethon, база знаний, инструкции и проверки остаются стандартными.
2. **Готовый контекст для ИИ**: Claude при формировании промптов видит жесткие правила в `passport.md` и автоматически учитывает их, обращаясь к полной инструкции в базе знаний при написании кода.
3. **Безопасность**: Защищает купленные аккаунты от случайного уничтожения из-за параллельных коннектов или отсутствия прокси.
