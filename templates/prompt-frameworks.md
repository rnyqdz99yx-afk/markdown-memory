# Prompt Frameworks для mm-bridge

> Структурные шаблоны для оборачивания промптов перед отправкой PowerShell-Клоду.
> Используются `mm-bridge --framework <name>` либо подбираются автоматом по типу задачи.
> Атрибуция: вдохновлено https://github.com/langgptai/awesome-claude-prompts (MIT).

## Когда какой использовать

| Тип задачи | Рекомендованный framework | Почему |
|---|---|---|
| Тривиальный фикс (1-2 файла) | **none** | Обёртка избыточна, просто прямой текст |
| Средняя задача с конкретным скоупом | **CRISPE** | Достаточно структуры, не громоздко |
| Сложная (рефакторинг, новая фича, ≥3 файлов) | **XML** | Явная разметка для модели = меньше галлюцинаций |
| Code review / архитектурный анализ | **PERSONA** | Роль направляет глубину анализа |
| Bug hunt / отладка | **HYPOTHESIS** | Заставляет формулировать гипотезы и тесты |

## Framework: CRISPE

**Capacity / Role · Insight · Statement · Personality · Experiment**

```markdown
# Capacity / Role
Ты — <роль>. <Уровень>.

# Insight
<Что Claude должен ЗНАТЬ перед началом — контекст, ограничения, предыдущие решения>

# Statement
<Конкретная задача в одном предложении-императиве>

# Personality
<Как отвечать: тон, длина, формат>

# Experiment
<2-3 альтернативных подхода с pros/cons либо «дай свою рекомендацию из A/B/C»>
```

Пример:
```markdown
# Role
Ты — Python backend разработчик с 10 годами опыта в aiogram.

# Insight
Бот @filtrator уже работает в проде. Структура: aiogram 3.x + SQLite через sqlmodel.
Менять архитектуру нельзя. Все хендлеры — async.

# Statement
Добавь команду `/stats` которая показывает кол-во сообщений за день из текущего чата.

# Personality
Краткие комментарии на русском только где критично. Никаких многословных docstring'ов.

# Experiment
Покажи свой подход одной строкой, перед тем как писать код.
```

## Framework: XML (рекомендуется для сложного)

Anthropic нативно понимает XML-tags — это дает точную ссылку на блоки в твоём ответе.

```markdown
<role>
Ты — <роль>. Работаешь в проекте <project_name>.
</role>

<context>
<stack>Python 3.12 · aiogram 3.x · SQLite · loguru</stack>
<conventions>
- Все хендлеры async
- Логирование через loguru, не logging
- Тексты UI на русском
</conventions>
<files_to_read_first>
- C:\path\to\passport.md (особенно секция 8)
- C:\path\to\src\handlers\base.py (паттерн всех хендлеров)
</files_to_read_first>
<dont_touch>
- src/core/router.py
- migrations/
</dont_touch>
</context>

<task>
<one_liner>Добавь команду /stats показывающую сообщения за день</one_liner>
<steps>
1. Создай хендлер в src/handlers/stats.py
2. Зарегистрируй в src/handlers/__init__.py
3. Тесты в tests/test_stats.py
</steps>
</task>

<constraints>
<must>
- Юзать sqlmodel, не raw SQL
- Эмодзи в ответе (📊)
- Покрытие тестами >80%
</must>
<must_not>
- Добавлять зависимости в pyproject.toml
- Менять схему БД
</must_not>
</constraints>

<done_when>
- [ ] /stats возвращает число > 0 если за день были сообщения
- [ ] /stats возвращает «Сообщений за день: 0» если не было
- [ ] pytest tests/test_stats.py зелёный
</done_when>

<output_format>
Сначала покажи план изменений (3-5 пунктов).
Жди подтверждения 'go'.
Потом пиши код.
В конце вызови /mm-save-session.
</output_format>
```

## Framework: PERSONA

Для review / архитектурных решений — задаёт «оптику» Клода.

```markdown
# Persona

Ты — <тип эксперта> с <X лет> опыта. Знаменит за <конкретное качество — например, «безжалостную простоту» как Linus, «явные контракты типов» как Hickey, «security-first» как OWASP>.

# Анализ должен идти послойно:
1. <Слой 1 — например, "data structures"> — что не так?
2. <Слой 2 — например, "сложность"> — как упростить?
3. <Слой 3 — например, "обратная совместимость"> — что сломается?
4. <Слой 4 — например, "real-world impact"> — кому нужно?
5. <Слой 5 — например, "вкус"> — это красиво?

# Объект анализа
<файл / PR / архитектура>

# Output
- Вердикт: ship / revise / reject
- Top-3 проблемы
- Top-1 рекомендация
```

Базируется на пятислойном фреймворке из `LinusTorvalds.md` (awesome-claude-prompts).

## Framework: HYPOTHESIS (отладка)

Заставляет Клода НЕ кидаться сразу фиксить, а сначала разобрать причину.

```markdown
# Симптом
<Что наблюдается. Stacktrace или поведение.>

# Окружение
<OS, версии, как воспроизвести>

# Что уже пробовали
<Список — чтобы Клод не повторял>

# Задача
1. Сформулируй 3 гипотезы причины (отсортируй по вероятности)
2. Для каждой — предложи минимальный эксперимент чтобы её подтвердить/опровергнуть
3. Жди моего «иди» с указанием какой эксперимент гонять
4. После эксперимента — отчёт + следующая гипотеза или фикс

# Запрещено
- Писать фикс до подтверждения гипотезы
- Менять > 1 переменной за раз
```

## Auto-detect (без явного флага)

Если `mm-bridge` вызван без `--framework`, выбирает по эвристикам из задачи:

```
содержит "ревью" / "посмотри" / "оцени" / "review"            → PERSONA
содержит "не работает" / "падает" / "bug" / "ошибка"           → HYPOTHESIS
длина задачи > 200 символов ИЛИ ≥3 шага в "Что нужно сделать"  → XML
длина задачи < 50 символов И один файл                         → none
иначе                                                          → CRISPE
```

Если эвристика неуверенна — спросить юзера: `Использовать XML / CRISPE / без обёртки? (XML по умолчанию)`.

## Что НЕ делать

- Не вкладывать один framework в другой (`<role>` внутри CRISPE).
- Не оборачивать тривиальные промпты — это раздражает.
- Не использовать английские теги в русских промптах если это не XML (XML — теги английские, контент русский).
- Не выдумывать новые «фреймворки» — этих 4 хватает на >90% случаев.
